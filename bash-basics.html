Command-Line Basics: Files, Pipes & Text

The two dozen commands that make the terminal genuinely useful — moving around, handling files, and the pipe trick that turns simple tools into a data-processing assembly line.

🟢 Beginner ⏱️ ~1.5 hr 💻 Bash

Open your terminal (set it up in the install lesson if you haven't). Type each command yourself — muscle memory is the whole game here.

1. Knowing where you are & moving around

The terminal always has a "current folder." Three commands are your eyes and feet:

pwd            # print working directory — where am I?
ls             # list what's in this folder
ls -l          # list with details (size, date)
ls -la         # also show hidden files (those starting with .)
cd Downloads   # change directory into Downloads
cd ..          # go up one level
cd ~           # go to your home folder

Decode the jargon: paths, . and ..

A path is an address for a file. . means "here" (current folder), .. means "the folder above," and ~ means "my home folder." Master these three and you'll never get lost.

2. Creating & handling files and folders

mkdir project        # make a new folder
cd project
touch notes.txt      # create an empty file
cp notes.txt copy.txt  # copy
mv copy.txt final.txt  # move OR rename
rm final.txt         # remove (careful — no recycle bin!)

⚠️ rm is forever

The command line has no trash can. rm deletes immediately and permanently. Read the line twice before you run it — especially before rm -r (which deletes whole folders). This caution is a habit worth building on day one.

3. Looking inside files

cat file.txt     # dump the whole file to the screen
head file.txt    # first 10 lines
head -n 3 file.txt  # first 3 lines
tail file.txt    # last 10 lines
less file.txt    # scroll through a big file (press q to quit)

head and less matter enormously in bioinformatics, where a single file can be gigabytes — you never want to cat a whole genome to your screen.

4. Searching & counting

grep "TP53" genes.txt   # print lines containing "TP53"
grep -c ">" seqs.fasta   # COUNT lines containing ">"
wc -l file.txt           # count lines in a file
sort names.txt           # sort lines alphabetically
sort names.txt | uniq    # sort, then remove duplicates

Why grep is a bioinformatics superpower

FASTA files mark each sequence with a header line starting >. So grep -c ">" instantly counts how many sequences are in a file of any size. Tiny command, huge time saver — you'll use it constantly.

5. The pipe | — the big idea

This is the concept that makes the command line powerful. The pipe | takes the output of one command and feeds it straight into the next, so small tools combine into something bigger.

# "Take the file, keep header lines, then count them"
cat seqs.fasta | grep ">" | wc -l

# "List files, find the .fasta ones, count them"
ls | grep ".fasta" | wc -l

Read a pipe left to right as a sentence: do this, then this, then this. Each tool does one small job well; the pipe chains them into an analysis.

6. Saving output with >

grep ">" seqs.fasta > headers.txt   # write results to a new file
grep ">" seqs.fasta >> headers.txt  # append to an existing file
✍️ Try it: make a file with echo "ATGC" > dna.txt, then check it with cat dna.txt. You just created and read a file entirely from the keyboard.

Your starter cheat sheet

pwdWhere am I?
ls / ls -lList files (with details)
cd folderMove into a folder (cd .. = up)
mkdir nameMake a folder
cp / mv / rmCopy / move-rename / delete
cat / head / tail / lessView a file's contents
grep "x" fileFind lines containing "x"
wc -l fileCount lines
sort / uniqSort lines / drop duplicates
cmd1 | cmd2Pipe: feed output into the next command
cmd > fileSave output to a file

Check your understanding

What does grep ">" seqs.fasta | wc -l tell you?

The number of sequences in the FASTA file — because each sequence has one header line starting with >, and you're counting those lines.

What's the difference between > and >>?

> creates (or overwrites) a file with the output. >> adds to the end of an existing file without erasing it. Mix them up and you can overwrite something you meant to keep — another reason to read before you run.

Why prefer head or less over cat for big files?

Because genomic files can be enormous. cat would flood your screen with millions of lines; head shows just the top, and less lets you scroll calmly.
